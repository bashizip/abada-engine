# Abada Engine 0.9.0 Release Notes

Release date: 2026-07-14

Abada 0.9.0 is the durable-runtime release. PostgreSQL is authoritative for
mutable workflow state, engine mutations are atomic, process instances remain
pinned to immutable definition versions, and lifecycle delivery uses a
transactional outbox.

## Highlights

- Flyway owns and validates the production schema through V6.
- Process definitions are immutable and versioned by deployment ID.
- Process instances, tokens, join state, variables, tasks, subscriptions,
  timers and external tasks recover from PostgreSQL after restart.
- Public mutation commands load and lock committed rows, validate, advance,
  persist resulting work, append history and commit as one transaction.
- External tasks and timer jobs use durable status, lease and retry state.
- Lifecycle history and outbox events commit with workflow state. Outbox
  dispatch uses PostgreSQL `FOR UPDATE SKIP LOCKED`, retry delays and stable
  event IDs; optional webhook targets receive `X-Abada-Event-Id`.
- Public task and instance queries are bounded and deterministically ordered.
- The supported BPMN subset has linked executable conformance evidence and a
  precise runtime semantics contract.

## Supported BPMN core

The guaranteed subset includes none start/end events, user tasks, embedded and
external service tasks, script tasks, exclusive/inclusive/parallel gateways,
and message/signal/duration-timer catch events. Event-based gateways are
limited to one outgoing catch event. Unsupported constructs are rejected at
deployment.

See the [BPMN support contract](../reference/bpmn-support.md) and
[runtime semantics](../reference/runtime-semantics.md).

## Database upgrade

Back up PostgreSQL before upgrading. Start exactly one 0.9.0 engine first and
allow Flyway to migrate the schema to V6; confirm application health and
`flyway_schema_history` before starting additional replicas. Automated
Testcontainers coverage verifies upgrades from every prior schema version
V1–V5.

Downgrading an upgraded database is not supported. Restore the pre-upgrade
backup when rollback is required.

## Verification evidence

- 101 backend tests passed on Java 21.
- 11 PostgreSQL restart, concurrency, transaction-boundary, durable-work and
  outbox acceptance tests passed against PostgreSQL 16.
- Five upgrade paths, V1 through V5 to V6, passed against PostgreSQL 16.
- Tenda and Orun remain on Node 24 and are covered by CI lint/build jobs.

## Known boundary

0.9.0 certifies the durable single-engine PostgreSQL runtime. Multi-replica
work acquisition, replica termination around commits, comprehensive
idempotency and rolling-upgrade certification remain the 0.10 release gate.
Embedded delegate side effects cannot be rolled back by PostgreSQL and must be
idempotent; external tasks are recommended for remote work.
