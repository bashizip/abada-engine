# Abada Engine 0.10.0-alpha Release Notes

Release date: 2026-07-19

Abada 0.10.0-alpha is the cluster-safety pre-release. It keeps PostgreSQL as
the authority for workflow state and adds database-coordinated work
acquisition, lease recovery, concurrent correlation and deterministic mutation
replay across engine replicas.

This remains an alpha release. REST and worker contracts are not frozen until
the 0.11 gate, and external side effects remain at-least-once.

## Highlights

- Timer jobs are claimed in bounded PostgreSQL `FOR UPDATE SKIP LOCKED`
  batches with durable 120-second leases and single-count attempts.
- External-task fetch-and-lock uses `SKIP LOCKED`, so concurrent workers and
  replicas receive disjoint work; expired locks remain recoverable.
- Message and signal subscriptions are consumed transactionally. Signal rows
  use stable lock ordering to avoid cross-replica ordering conflicts.
- Task and event advancement reject terminal process instances after locking
  authoritative process state, preventing cancellation races from resurrecting
  workflow progress.
- Every public mutation endpoint accepts an optional `Idempotency-Key`.
  PostgreSQL atomically reserves the key with the workflow command and replays
  its stored response for matching duplicate requests.
- Transactional-outbox dispatchers acquire independent batches with stable
  event IDs for downstream deduplication.
- V8 adds non-destructive indexes for timer and external-task acquisition.
- Deployment compiles standard BPMN, Abada-native and Camunda 7 assignment
  dialects into one canonical model, persists compatibility reports, and
  exposes strict validation and deterministic Camunda-to-Abada migration.

## Database upgrade

Back up PostgreSQL before upgrading. Start one 0.10.0-alpha replica and allow
Flyway to migrate the 0.9.0-alpha V6 schema through V7 and V8. V7 adds
definition compatibility metadata and task assignment strategy without
discarding existing rows. V8 adds three acquisition indexes:

- `idx_jobs_available_acquisition`
- `idx_jobs_expired_lease_acquisition`
- `idx_external_tasks_acquisition`

No table or application-data shape changes are made. Confirm application
health and schema version V8 before starting the remaining replicas. Automated
PostgreSQL Testcontainers coverage verifies upgrades from V1 through V7.
Downgrading the schema is not supported; restore the pre-upgrade backup if a
rollback is required.

## Cluster semantics

Exactly-once refers to a committed workflow-state transition. A replica may
die after an external system accepted a side effect and before acknowledgement,
so workers, webhook consumers and outbox consumers must deduplicate with their
protocol identifier or Abada event ID.

The supported BPMN subset has no separate asynchronous-continuation job type.
Synchronous continuations execute inside their owning atomic command; timers
and external tasks are the durable asynchronous boundaries.

## Verification evidence

- 137 backend tests pass on Java 21.
- 21 restart, atomicity and multi-replica tests run two application contexts
  against one PostgreSQL 16 Testcontainer for
  timers, external tasks, user tasks, messages, signals, idempotency records
  and outbox records.
- Replica termination with an owned timer lease, failures before and after
  transaction commit, duplicate requests and cancellation/correlation races
  are covered by executable acceptance tests.
- Seven supported schema upgrade paths, V1 through V7 to V8, are exercised.
- Tenda lint/build and Orun build pass on Node 24; both npm audits report zero
  vulnerabilities.
- The executable Spring Boot JAR, production Docker image and both production
  Compose configurations pass release verification.
